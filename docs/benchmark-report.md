# RPC Benchmark 测试报告

## 测试环境

- **OS**: Ubuntu 24.04 (WSL2)
- **测试日期**: 2026-04-17
- **服务端**: `rpc_server` (多线程模式, `start_multi`)
- **客户端**: `rpc_benchmark`
- **测试方法**: 调用 `SystemMonitorService.GetStatus`
- **网络**: 本机回环 (127.0.0.1:12345)

## 测试设计

公平对比，**两种模式都是单线程**：

| 模式 | 描述 |
|------|------|
| **协程并发** | 1 线程 + N 协程，所有协程 pin 到同一个 Processor，通过 epoll 事件驱动实现并发 IO |
| **同步串行** | 1 线程，阻塞式 socket，串行发送请求 |

## 结论

### 协程的优势：吞吐量

在**同样单线程**的条件下，协程版本的 QPS 稳定提升 **48-60%**，总完成时间缩短 **32-37%**。

原因：同步版在等待网络 IO 时线程完全空转，而协程版在等待时切换到其他协程继续处理，IO 等待时间被有效重叠。

### 协程的延迟特性

协程版的单请求平均延迟更高，这是并发系统的固有特征：

- 同步版延迟 = 纯网络 RTT（~130us）
- 协程版延迟 = 网络 RTT + 排队等调度时间

N 个协程共享 1 个线程，每个协程在 IO 就绪后需要等待调度器切换回来，排队时间与协程数成正比。这不是缺陷，而是**用单请求延迟换取整体吞吐量**的并发设计权衡。

### 核心价值

协程让单线程具备了并发 IO 能力，无需创建大量线程即可获得显著的吞吐量提升，同时避免了：
- 内核态线程上下文切换开销
- 每线程 MB 级栈内存占用
- 线程数受限于系统资源的瓶颈
